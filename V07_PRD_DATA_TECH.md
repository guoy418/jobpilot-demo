# JobPilot v0.7 融合版说明

## 定位

v0.7 以 v0.4 的页面结构和视觉节奏为基底，吸收 v0.6 的“模块内创建正式记录”逻辑。目标是做一个更轻、更清楚的产品逻辑原型：用户在岗位、面试、答案库、简历版本等具体模块里新增或上传内容，满足最小必填字段后直接生成正式记录，再由系统把这些模块里的下一步动作收束到今日待办。

## 主链路

1. 用户进入具体模块。
2. 点击该模块的“上传 / 新增”按钮。
3. 第一步先选择原始材料，或粘贴文字内容。
4. 第二步由系统模拟解析为结构化字段，用户补齐必填字段和可选备注。
5. 创建正式记录，不再进入统一材料收件箱确认。
6. 今日待办从正式记录中自动汇总：待投递/跟进岗位、待复盘面试、薄弱回答练习、训练计划动作。

## 废弃路线

旧路线“全局导入 -> 材料收件箱 -> 确认草稿 -> 写入模块”在 v0.7 中不作为主链路。材料收件箱代码暂时保留为历史/临时暂存候选，但不出现在主导航，也不参与今日待办计算。后续如果需要恢复，只能作为“不确定归属材料”的临时区，而不是所有材料的必经确认环节。

## 数据模型边界

- Opportunity：正式岗位记录，包含 JD 原文、原材料、状态、优先级、匹配度、使用简历、时间线、下一步动作。
- InterviewSession：一场面试复盘，包含关联岗位、原录音/文字稿、多个 QA Pair。
- AnswerCard：可复用回答，来源可以是面试复盘或手动创建。
- ResumeVersion：上传过的简历文件版本，岗位详情记录实际投递使用哪一版。
- TrainingPlan/WeeklyPlan：周目标、重点方向、泛任务训练和手动计划动作。具体岗位和具体面试问题不在这里重复录入。
- TodayAction：不独立存库，应该由 Opportunity / InterviewSession / AnswerCard / WeeklyTask 派生。

## 自动与手动

- 自动：今日待办汇总、投递进度统计、面试薄弱回答统计、岗位时间线追加、模块间跳转、训练任务来源跳转。
- 手动：选择文件、补齐必填字段、选择简历版本、确认主观优先级/下一步动作、设置泛任务优先级、编辑复盘和答案卡。
- 半自动：匹配度、JD 摘要、岗位字段抽取、录音转文字稿、面试 QA 拆分、答案优化，后续接 AI 后生成，但用户仍可编辑确认。

## 模块内上传规则

- 岗位管理：原始材料可以是 JD 截图、PDF、文字 JD 或招聘链接。系统解析公司、岗位、城市、匹配度等字段；用户确认简历版本、优先级和下一步动作。
- 面试复盘：原始材料可以是面试录音、录音转写稿或文字稿。若是录音，真实版本先转文字稿，再识别问题、原回答、评价和优化建议；用户补充关联岗位和备注。
- 面试复盘里的 `weak` QA 是今日待办的来源。用户应能逐题标记“已处理”或重新标为薄弱，而不是只能整场面试一起清理。
- 简历版本：原始材料是简历文件。系统解析版本名称、方向、核心卖点和摘要；岗位实际使用哪份简历由岗位记录关联，不在简历页重复维护。

## 岗位生命周期

- 岗位详情右侧的主视图不是材料录入日志，而是固定生命周期：待投递 -> 已投递 -> 笔试 -> 约面 -> 等结果 -> Offer。
- “笔试”是可选节点，没有相关信息时可以显示为跳过，不阻塞约面或等结果。
- 系统会根据正式记录自动推进：新增 JD 后默认为待投递；点击已投递后进入已投递；新增并关联面试复盘后进入约面。
- 用户可以手动覆盖进度，手动覆盖会写入系统日志，但不删除原始材料和历史动作。
- 后端不要把 TodayAction 作为单独事实表；今日待办应由岗位当前状态、下一步动作、面试复盘和训练计划派生。
- 岗位 P 级不是状态死映射：应综合状态、`deadline`/`dueDate`、匹配度和主观优先级计算；训练/杂项任务由用户手动设置 P0-P3。

## 当前实现状态

- v0.4 页面结构保留：岗位管理看板、面试复盘、答案库、简历版本、训练计划、设置导出。
- 主导航移除材料收件箱。
- 今日待办不再读取 Inbox 草稿。
- 岗位 / 面试 / 答案 / 简历新增入口改为模块内弹窗。
- 当前已进入本地单用户 MVP：前端可从本地 API hydrate，后端使用 SQLite 持久化，文件存储在本地目录；公开 demo 仍使用静态 mock 数据。
- 数据管理已包含完整 JSON 备份/恢复，备份包含本地存储文件的 base64 内容。
- 解析接口已接入本地文件提取和可选 AI provider：可本地读取粘贴文本、txt/md、文字型 PDF、DOCX；截图 OCR 和录音转写通过 Assist/provider 配置挂在现有 parse API 边界后面。解析失败会返回 `extractionStatus`，前端会阻止进入 review 并显示原因。
- 设置里有两个独立开关：**文字/JD/简历解析** 控制 JD/简历解析和面试文字稿 AI 复盘；**录音转写** 只控制无粘贴文字时的音频转写。
- 面试复盘 Local 模式：粘贴/上传文字稿 -> 本地规则拆题 -> 用户确认 -> 创建正式记录，不调用模型。
- 面试复盘 Assist 模式：后端两阶段 fan-out。先整稿抽 Q/A，再逐题生成评分、评价、回答框架、可背版本；失败时返回 `ai-parser-failed` / `ai-review-empty`，不会静默写回旧规则复盘内容。
- 面试复盘解析会返回结构化 `qaPairs`；Assist 成功时使用 API 返回内容。没有真实文字稿时不再生成假的默认问题。
- 面试原始材料支持应用内预览文字稿/转写内容；source file 内容会随面试记录持久化。
- 答案卡已接入训练计划：答案卡可一键生成训练任务；答案来源训练任务完成后，会把答案卡回写为 `ACTIVE` / `可复用`。
- 答案卡状态为 `NEEDS PRACTICE` 时可直接派生 Today Todo；若已有打开的答案关联训练任务则不重复生成。
